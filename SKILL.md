# SKILL.md — Karpathy Guidelines (동네대전 적용판)

> Andrej Karpathy의 LLM 코딩 실수 방지 지침을 기반으로,
> 동네대전 프로젝트 컨텍스트에 맞게 작성한 행동 지침이다.
> 루트에 하나만 존재하며, 모든 서비스에서 공통으로 참조한다.
>
> 원문 출처: https://x.com/karpathy/status/2015883857489522876

---

**트레이드오프:** 이 지침은 속도보다 신중함을 우선한다. 사소한 작업에는 판단을 사용한다.

---

## 1. 코딩 전에 생각하기

**가정하지 말 것. 혼란을 숨기지 말 것. 트레이드오프를 드러낼 것.**

구현 전에:
- 가정은 명시적으로 서술한다. 불확실하면 질문한다.
- 해석이 여러 가지라면 제시한다. 조용히 하나를 고르지 않는다.
- 더 단순한 접근이 있으면 말한다. 필요하면 반박한다.
- 불명확한 것이 있으면 멈춘다. 무엇이 혼란스러운지 명시하고 질문한다.

## 2. 단순하게 먼저

**문제를 해결하는 최소한의 코드. 투기적인 것은 없다.**

- 요청받지 않은 기능은 추가하지 않는다.
- 단일 용도 코드에 추상화를 넣지 않는다.
- 요청받지 않은 유연성이나 확장성을 추가하지 않는다.
- 불가능한 시나리오에 대한 에러 처리는 하지 않는다.
- 200줄로 작성했는데 50줄로 가능하다면 다시 작성한다.

스스로에게 묻는다: "시니어 엔지니어가 이걸 보고 과복잡하다고 할까?" 그렇다면 단순화한다.

## 3. 외과적 변경

**반드시 필요한 것만 건드린다. 자신이 만든 것만 정리한다.**

기존 코드 편집 시:
- 인접한 코드, 주석, 포맷을 "개선"하지 않는다.
- 망가지지 않은 것은 리팩토링하지 않는다.
- 기존 스타일을 따른다. 본인이 다르게 하고 싶어도.
- 관련 없는 데드 코드를 발견하면 언급만 한다. 삭제하지 않는다.

내 변경이 고아를 만들면:
- 내 변경으로 인해 사용되지 않는 import/변수/함수를 제거한다.
- 기존에 있던 데드 코드는 요청받지 않으면 건드리지 않는다.

검증: 변경된 모든 줄이 사용자의 요청으로부터 직접 추적되어야 한다.

## 4. 목표 중심 실행

**성공 기준을 정의하고, 검증될 때까지 반복한다.**

작업을 검증 가능한 목표로 변환한다:
- "유효성 검사 추가" → "잘못된 입력에 대한 테스트 작성 후 통과시키기"
- "버그 수정" → "버그를 재현하는 테스트 작성 후 통과시키기"
- "리팩토링" → "전후로 테스트 통과 확인"

다단계 작업 시 간략한 계획을 서술한다:
```
1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
3. [단계] → 검증: [확인 방법]
```

강한 성공 기준은 독립적으로 반복을 가능하게 한다.
약한 기준("되게 만들어")은 지속적인 명확화를 필요로 한다.

---

**이 지침이 잘 작동하고 있다면:**
diff에서 불필요한 변경이 줄고, 과복잡화로 인한 재작성이 줄고,
실수 후가 아니라 구현 전에 명확화 질문이 나온다.
