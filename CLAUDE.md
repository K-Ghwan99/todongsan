# CLAUDE.md — 동네대전 공통 가이드

> 모든 서비스에 적용되는 공통 행동 지침이다.
> 각 서비스 디렉토리의 `CLAUDE.md`는 이 문서를 상속하며, 서비스별 맥락을 추가한다.

---

## 0. 코드 작성 전 필수 문서 읽기

**작업 전 반드시 읽어야 할 문서 목록과 우선순위:**

```
1순위 (항상 읽기)
├── README.md                  — 서비스 전체 개요, 핵심 설계 원칙, 사용자 흐름
├── CONVENTION.md              — 패키지 구조, 네이밍, 응답 포맷, Enum 정책
├── ERROR_POLICY.md            — 공통 에러 코드, HTTP 상태 코드 매핑 기준
└── ERD.md                     — 전체 서비스 간 관계 개요

2순위 (내 서비스 작업 시)
├── docs/{my-service}/ERD.md           — 내 서비스 최신 ERD (버전 번호 없는 파일)
├── docs/{my-service}/API_SPEC.md      — 내 서비스 최신 API 명세
└── docs/{my-service}/ERROR_CODE.md   — 내 서비스 도메인 에러 코드

3순위 (연계 API 작업 시)
└── docs/{target-service}/API_SPEC.md  — 호출하거나 호출받는 서비스 명세만
```

**버전 파일 읽기 규칙:**
- `ERD.md`, `API_SPEC.md`처럼 버전 번호가 없는 파일이 존재하면 그것이 최신본이다.
- 없으면 `_ver{숫자}` 중 숫자가 가장 큰 파일을 읽는다.
- 구버전 파일(`ERD_ver1.md` 등)은 읽지 않는다. 변경 이력은 최신 파일 내 "변경 내역" 섹션을 참조한다.

## 1. 코딩 전 생각하기

**가정하지 말고, 혼란을 숨기지 말고, 트레이드오프를 드러낸다.**

구현 전에:
- 가정은 명시적으로 서술한다. 불확실하면 질문한다.
- 해석이 여러 가지라면 제시한다. 조용히 하나를 고르지 않는다.
- 더 단순한 접근이 있으면 말한다. 필요하면 반박한다.
- 불명확한 것이 있으면 멈춘다. 무엇이 혼란스러운지 명시하고 질문한다.

**이 프로젝트에서 특히 주의할 것:**
- 서비스 간 REST 호출 시, 상대 서비스의 API_SPEC을 먼저 확인한다.
- DB 스키마 변경 전, ERD.md에 정의된 UNIQUE 제약과 인덱스를 확인한다.
- Point 관련 로직은 멱등성 키(`Idempotency-Key`) 처리를 항상 고려한다.

## 2. 단순하게 먼저

**문제를 해결하는 최소한의 코드. 투기적인 것은 없다.**

- 요청받지 않은 기능은 추가하지 않는다.
- 단일 용도 코드에 추상화를 넣지 않는다.
- 요청받지 않은 유연성이나 확장성을 추가하지 않는다.
- 불가능한 시나리오에 대한 에러 처리는 하지 않는다.
- 200줄로 작성했는데 50줄로 가능하다면 다시 작성한다.

**이 프로젝트에서 특히 주의할 것:**
- MVP 범위(README.md 섹션 16)를 벗어난 기능은 구현하지 않는다.
- `△` 표시 기능은 팀 결정 전까지 구현하지 않는다.
- Kafka, MCP, 이벤트 브로커는 MVP 제외 항목이다. 구조만 고려하고 구현하지 않는다.

## 3. 외과적 변경

**반드시 필요한 것만 건드린다. 자신이 만든 것만 정리한다.**

기존 코드 편집 시:
- 인접한 코드, 주석, 포맷을 "개선"하지 않는다.
- 망가지지 않은 것은 리팩토링하지 않는다.
- 기존 스타일을 따른다. 본인이 다르게 하고 싶어도.
- 관련 없는 데드 코드를 발견하면 언급만 한다. 삭제하지 않는다.

내 변경이 고아를 만들면:
- 내 변경으로 인해 사용되지 않는 import/변수/함수를 제거한다.
- 기존에 있던 데드 코드는 요청받지 않으면 건드리지 않는다.

**이 프로젝트에서 특히 주의할 것:**
- DTO 필드 변경 시 해당 서비스 담당자 외에 팀 전체에 공유한다 (README.md 섹션 19-2).
- 공통 응답 포맷(`ApiResponse`)은 CONVENTION.md 기준에서 임의로 변경하지 않는다.
- 에러 코드 추가 시 `ERROR_POLICY.md`(루트)와 해당 서비스 `ERROR_CODE.md` 양쪽에 반영한다.

## 4. 목표 중심 실행

**성공 기준을 정의하고, 검증될 때까지 반복한다.**

작업을 검증 가능한 목표로 변환한다:
- "유효성 검사 추가" → "잘못된 입력에 대한 테스트 작성 후 통과시키기"
- "버그 수정" → "버그를 재현하는 테스트 작성 후 통과시키기"
- "리팩토링" → "전후로 테스트 통과 확인"

다단계 작업 시 간략한 계획을 서술한다:
```
1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
3. [단계] → 검증: [확인 방법]
```

**이 프로젝트에서 공통 성공 기준:**
- 애플리케이션이 정상 기동된다 (`/actuator/health` 200 OK).
- 공통 응답 포맷이 CONVENTION.md와 일치한다.
- 에러 응답에 `errorCode`, `message`가 포함된다.
- 서비스 간 REST 호출이 명세된 엔드포인트와 일치한다.

## 5. MSA 협업 원칙

**서비스 경계를 침범하지 않는다.**

```
각 서비스는 자신의 DB만 직접 접근한다.
타 서비스 데이터가 필요하면 반드시 REST API로 요청한다.
타 서비스 테이블을 직접 JOIN하거나 참조하지 않는다.
```

**서비스 간 통신 원칙 (README.md 섹션 5-3):**
- 모든 요청/응답은 공통 응답 포맷(`ApiResponse<T>`)을 따른다.
- Point 관련 API는 멱등성 키를 사용한다.
- 이벤트 브로커는 사용하지 않는다. REST 직접 호출만 한다.

**타 서비스 ID 참조 방식:**
- 외래 서비스 ID(`member_id`, `battle_id`, `market_id`)는 FK 없이 `BIGINT` 컬럼으로만 보관한다.
- DB 레벨 JOIN은 없다. 데이터 조합은 서비스 레이어에서 REST 호출로 처리한다.

## 6. Git 협업 규칙 (README.md 섹션 19)

```
직접 main push 금지
feature/* → develop PR → 리뷰 1명 이상 → merge
API 응답 포맷/DTO 변경 시 팀 전체 공유
매일 develop 최신화
```

**커밋 컨벤션:**
```
feat:     기능 추가
fix:      버그 수정
refactor: 리팩토링
docs:     문서 수정
test:     테스트 코드
chore:    설정 수정
```

## 부록: 서비스별 CLAUDE.md 작성 안내

각 서비스 디렉토리의 `CLAUDE.md`는 이 루트 문서를 참조하며 다음을 추가한다.

```markdown
## 내 서비스 컨텍스트
- 담당 서비스명, 패키지 루트
- 핵심 도메인 테이블 목록
- 주요 비즈니스 제약 요약

## 연계 서비스
- 내가 호출하는 서비스와 엔드포인트
- 나를 호출하는 서비스와 엔드포인트

## 이 서비스의 성공 기준
- 서비스별 검증 포인트
```
